问题分析：
请求“查看ptz信息”已经被 RAG 判到 agent=ptz，但因为 score=0.00 被判成 confidence=low，系统继续走 Router LLM；Router LLM 请求超过 60 秒被中断，LangSmith 报 AbortError，于是整体体验变成“卡 60 秒再 fallback”。核心问题是：ptz 这类明确意图没有被“快速直达”，反而被送去慢路径 LLM。
日志链路解释（对应你贴的日志）

[DualKBRoute] ... Agent: ptz | Time: 5418ms
RAG 规则里已经把它判为 ptz，但这里的分数还是 0。
[Route] ... confidence=low, score=0.00
confidence 低 → 触发 Router LLM 兜底。
[Router] Starting LLM route ...
进入 LLM Structured Output 路由。
60 秒后 Fallback: using RAG agentType=ptz
LLM 侧请求超时或被 abort，LangSmith 报 AbortError。
为什么会“明明识别了 ptz 还要走 LLM”
目前路由逻辑是：

RAG confidence != low 才直接走路由。
你这个 query 的 topScore=0.00 → confidence=low → 必走 LLM。
LLM 超时后才回退到 RAG ptz。
所以“ptz 明确意图”被错误地拖到慢路径。

问题分析：
从日志可以看出，用户发送"查看ptz信息..."命令后，系统：
正确识别到了PTZ意图（ptz_camera得分0.90）
路由到了PtzAgent节点
PtzAgent开始处理，构建了上下文并获取了2个PTZ工具
但在LLM调用阶段发生了超时中断（AbortError）

根因：上游超时/取消信号触发 → LangGraph 把 signal 传给子 runnable → ChatOpenAI 调用被 abort → 抛 AbortError

核心问题定位：
你这次从 23:38:24 → 23:40:09 大约 105 秒才 abort，说明 “60秒超时”不一定是唯一的 abort 来源；但无论谁触发 abort，你当前代码的问题是：
ptzAgent 对 AbortError 没有“区分处理”
LLM 调用没有设置一个“明确、可控的请求超时”（只靠外层 abort，体验会很差：卡很久然后炸）
Graph 的 timeout 写死 60s，不可配置（不同环境/模型延迟不同）

让系统在 LLM/网络慢的时候做到：
1.可控超时：例如 25~45 秒内给出“正在处理/稍后再试”的稳定响应，而不是卡 100 秒再炸
2.错误可解释：AbortError 不再只打一个 error log，而是告诉你“谁触发了取消、耗时多久、当时在做什么”
3.可配置：通过环境变量/配置调整 timeout，而不是改代码
