以下为**仅评价、不修改**的 code review 结果，按严重度排序。若无改动需求，我不会触碰代码。

**高风险**
1. **RAG 路由无法稳定命中 PTZ（仅靠显式关键字）**  
`src/chatbot/knowledge_base/knowledge.service.ts`  
当前 `AgentType` 只基于 control/vision 双 KB 计算，`ptz` 不在 DualKB 逻辑里，只有 `hasPtzWord` 的显式规则才会路由到 `ptz`。这意味着“PTZ 状态/能力”类请求如果没有命中关键词（或被同义词替换/非中文表述），可能仍被路由到 `home_control/vision/general`，工具门控也可能不给 PTZ 工具。  
风险：PTZ 场景召回不稳定。

**中风险**
1. **Digest 认证实现较“最小”，兼容性不完整**  
`src/http-client/ptz-http-client.service.ts`  
当前只支持 `MD5`，未处理 `MD5-sess` / `stale=true` / `qop` 多值 / `auth-int` 等情况；若设备返回不同 Digest 参数，可能无法鉴权。  
风险：部分机型或固件版本调用失败。

2. **PTZ 消息上下文未独立存储**  
`src/chatbot/langgraph/state.ts`, `src/chatbot/langgraph/muti-agent-graph.ts`  
PTZ agent 的 `ptzMessages` 没接入（你为避免编码问题移除了），因此 PTZ 对话不会像 Vision/HomeControl 一样保留独立上下文。  
风险：多轮 PTZ 对话的上下文记忆弱。

3. **工具执行时 XML 解析不足**  
`src/http-client/ptz-http-client.service.ts`  
PTZ ISAPI 常返回 XML。当前返回 JSON 包装里 `data` 可能是原始 XML 字符串。对“相似写法”的要求已满足，但若后续 LLM 需要提取字段，解析难度较高。  
风险：下游解析成本较高。

**低风险**
1. **默认凭据写死在配置**  
`src/config/app.config.ts`  
PTZ 的默认 `url/username/password` 写入了真实值。  
风险：配置泄漏或误部署到其他环境。

2. **编码/乱码迹象**  
`src/chatbot/knowledge_base/knowledge.service.ts`, `src/chatbot/knowledge_base/tools_v2.json`  
从内容检索结果看，部分中文已出现 mojibake（如 `äº‘å°`）。这可能是文件编码混乱导致。  
风险：日志/提示词可读性下降，规则匹配可能失效。

3. **RAG 工具库统计字段手改**  
`src/chatbot/knowledge_base/tools_v2.json`  
`statistics.total_tools` 手动修改，易与实际工具数量不一致。  
风险：仅限统计错误，不影响功能。
