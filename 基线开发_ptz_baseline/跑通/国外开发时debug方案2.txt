结论先说：**PTZ agent 链路“部分打通”**，目前卡在“参数被拆分丢失”，所以没有触发 `ptz_get_status`，更没有进入 PTZ HTTP 调用层。

1. 现象与根因
1. 你的原始命令是 `查询当前ptz状态，通道id为1`，被预处理按逗号拆成两段：`查询当前ptz状态` + `通道id为1`。这在日志里已出现。
2. 代码里确实把逗号当“显式复合词”并参与拆分，见 [command-preprocessor.ts:44](/opt/ai/src/chatbot/preprocessor/command-preprocessor.ts:44) 和 [command-preprocessor.ts:52](/opt/ai/src/chatbot/preprocessor/command-preprocessor.ts:52)。
3. 拆分后，`ContextBuilderFactory` 对每个子句单独路由；`通道id为1` 这种参数片段很容易被判成 `general`，且默认会走 general fallback，见 [context-builder.factory.ts:335](/opt/ai/src/chatbot/context-builder/context-builder.factory.ts:335)。
4. 因为出现了 `ptz + general` 两个意图，被当成 compound，`ptzAgent` 只拿到子命令 `查询当前ptz状态`，见 [multi-agent-graph.ts:1175](/opt/ai/src/chatbot/langgraph/multi-agent-graph.ts:1175)。
5. PTZ prompt 明确写了“没 channelId 就反问”，见 [ptz-context.builder.ts:22](/opt/ai/src/chatbot/context-builder/ptz-context.builder.ts:22)，所以模型输出“请提供通道ID”是符合当前设计的。
6. `toolCalls=0`，没有 `tool used: ptz_get_status`，说明工具层未触发；自然也不会到 [ptz-camera.tool.provider.ts:125](/opt/ai/src/chatbot/tools/ptz-camera.tool.provider.ts:125) 和后续 HTTP client。

1. `router` 节点：先看输出里的 `isCompound/intents/subCommand`，确认是否被拆成 `ptz + general`，以及 `ptz` 子命令是否丢了 `channelId`。
2. `compoundExecutor` 节点：看每个 agent 的输入命令，重点是 `ptzAgent using subCommand`。
3. `compoundExecutor-ptzAgent-llm` 子运行：看 messages（user 最后一条是否只有“查询当前ptz状态”）和 `tool_calls`（此处应为空）。

5. 你这次截图里只有 `compoundExecutor-ptzAgent-llm`，没有 PTZ 工具节点，已经直接证明“未触发工具”。

2. 未打通：ptzAgent-llm -> `ptz_get_status` tool -> `PtzHttpClientService` 外部 PTZ API。
3. 判定依据：`toolCalls=0` + 无 `tool used: ptz_get_status` 日志。

4. 解决方案方向（仅分析，不改代码）
1. 拆分策略收敛：不要把“逗号+参数补充句”（如 `通道id为1`）当独立意图；应并入前一句同 agent 命令。
2. 参数继承机制：在 explicit compound 后处理阶段，识别参数片段（channelId、deviceId、location 等）并回填给前一个非 general intent。
3. PTZ agent 兜底提参：即使进入 compound，也可从原始整句或对话上下文提取 `channelId`，避免直接反问。
4. 增加回归用例：`查询当前ptz状态，通道id为1`、`查云台能力，通道2`、`查询ptz状态，1号通道`，断言必须产生 `ptz_get_status(channelId=1)`。

    ragRouteResult: ragResult,
  });
}
原始代码块引发的问题：
当子句是“状态如何？通道是1”这类“参数补充句”时，agentType 常为 null，被直接降级成 general，导致 channelId 与 PTZ 子句分离，最终出现 general + ptz + general 噪声结构。

解决方案（最简洁版本）：


改动内容1--“通道参数补充句”作为ptz子命令，而不是降级为general

可以，给你“最简单先跑通版”，只改 **1 个文件 1 处逻辑** 就够了。

1. 改动文件  
[context-builder.factory.ts](/opt/ai/src/chatbot/context-builder/context-builder.factory.ts)

2. 改动位置  
`handleExplicitCompound(...)` 里，`intents` 构建完成后、`agentTypes` 判定前。

3. 直接加这段最小逻辑（把“通道=1”这类 general 子句并回 ptz）
```ts
// 最小兜底：如果有 PTZ 意图，把“通道参数句”从 general 并入 PTZ
const channelPattern =
  /(通道\s*(id|号)?\s*(为|是|=|:)?\s*\d+|\d+\s*号?\s*通道|channel\s*(id)?\s*(is|=|:)?\s*\d+)/i;

const ptzIdx = intents.findIndex((i) => i.type === "ptz");
if (ptzIdx >= 0) {
  for (let i = intents.length - 1; i >= 0; i--) {
    if (i === ptzIdx) continue;
    const isGeneral = intents[i].type === "general";
    const sub = intents[i].subCommand || "";
    if (isGeneral && channelPattern.test(sub)) {
      intents[ptzIdx].subCommand =
        `${intents[ptzIdx].subCommand || ""} ${sub}`.trim();
      intents.splice(i, 1);
    }
  }
}
```

4. 为什么这是最简单且有效  
- 不改现有路由大框架。  
- 只修你当前核心问题：`channelId` 被拆到 general。  
- 改完后 PTZ 子命令会带上“通道是1”，更稳定触发 `ptz_get_status / ptz_get_capabilities`。  
- 还能减少你看到的 `general + ptz + general` 噪声组合。

如果你愿意再加“第二个最小改动”，我建议再在 `compoundAggregator` 过滤掉明显闲聊型 `generalAgent` 响应。


改动内容2--“问候句"作为噪声句应该被过滤掉

根因分析：“噪声问题”大概率还没完全解决
你的输入里有 你好kon，这段仍会保留为 general intent（补丁只并了“通道参数句”，没并“问候句”）。
compoundAggregator 目前只过滤很少几类默认文案，中文寒暄噪声基本不会被过滤：

解决方案：src/chatbot/langgraph/multi-agent-graph.ts中的私有类mergeAgentResponses
const responses: string[] = [];

for (const result of agentResults) {

总结：
结论：**对你当前主诉场景，噪声问题已“基本充分解决”，但不是理论 100% 全场景解决。**

基于你现有改动的静态判断：

1. 参数补充句噪声  
- 已处理：`general` 里的“通道是1”会并入 `ptz` 子命令。  
- 对你的句子 `你好kon，我想知道当前云台有什么功能，状态如何？通道是1`，这部分能生效。

2. 问候句噪声  
- 已处理：`mergeAgentResponses` 在存在 PTZ 工具结果时，会过滤 `generalAgent` 的常见问候/引导文案。  
- 你这类“你好kon + PTZ请求”大概率不会再把问候混到最终回复里。

3. 剩余边界（仍可能有少量噪声）  
- 如果 `general` 产出的噪声文案不在你当前正则关键词里，可能漏过滤。  
- 如果 PTZ 本轮没有产生 `toolResults`（例如模型没调工具/工具异常），过滤条件不触发，`general` 噪声可能回流。

最终判断：  
- **就你当前问题与用例，已经足够上线验证。**  
- 若要“彻底无噪声”，下一步是扩充 `general` 噪声正则或改成“当存在 ptz intent 时默认降权/忽略 general 输出”。
