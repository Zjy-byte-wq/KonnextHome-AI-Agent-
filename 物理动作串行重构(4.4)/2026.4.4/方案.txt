####### 现存问题的系统性分析 #########
<1>现象理解：
当前系统把“执行顺序约束”放在了 prompt 里，但没有放进执行引擎里。因此只要 LLM 在同一轮里产出多个 PTZ tool call，LangGraph 就会把这些工具节点并发放开，导致所有“先动云台、再缩放、再截图”这类链式动作失效。7 组测试都出现“工具几乎同时启动”，这不是偶发，而是同一个架构缺陷在不同指令上的重复表现。
<2>根因分析：
为什么说是系统性问题
根因不在单个工具，而在调度层设计。

ptzAgent 会一次性产出多个 tool calls
见 multi-agent-graph.ts (line 640)。

routeFromPtz 会把这些 tool call 映射成多个节点并一起返回
见 multi-agent-graph.ts (line 682)。
它返回的是 string[]，例如：
["ptzMoveToPreset", "ptzZoom", "ptzCaptureFrame"]

LangGraph 对 addConditionalEdges(...)-> string[] 的语义就是多分支并发
图定义在 multi-agent-graph.ts (line 168)；
各 PTZ 节点又都直接连到 aggregateResults，见 multi-agent-graph.ts (line 185)。

你虽然在 prompt 里写了 “Physical actions must be serial”，但执行器没有遵守这个约束
见 ptz-context.builder.ts (line 31)。

所以问题本质是：
语义层要求串行
调度层默认并行
运行时没有依赖管理

更深一层的问题
这还暴露出两个设计风险：

顺序依赖靠 LLM 自觉，不可靠
LLM 可以理解“先后”，但执行引擎没有 dependency model，结果仍会并发。

工具节点粒度不对
move_to_preset_tool、camera_zoom、capture_frame_tool 被拆成独立节点后，如果没有串行执行器，就天然容易被图引擎并发调度。


########### 解决方案分析###########
我建议的最终方案
按优先级：

先做方案 A
把 PTZ 工具执行改成单执行节点顺序跑，立即修复线上行为。

再做方案 B
补执行策略元数据，把“串行/并行”从 prompt 约束升级成 runtime 约束。

对高频复合意图补方案 C
把最常见的 PTZ 多步操作做成组合工具，减少 LLM 自由编排。

为什么不建议只改 prompt
因为你已经验证了：prompt 明确写了“must be serial”，但系统仍并发。
这说明问题不在理解层，而在执行层。继续调 prompt 只会让问题更隐蔽，不会真正消失。

一句话根因
当前架构把 PTZ 多步动作当成“多个可并发图节点”处理了，但 PTZ 实际上是“共享单一物理资源的状态机”，这两者模型不匹配。

如果你要，我下一步可以直接给你一版可落地的改造方案：
把 routeFromPtz 改成单节点 ptzToolExecutor，并给出需要修改的具体代码位置。



################ 最终落地方案 #############
把 PTZ 多步动作从“LLM 产出多个工具后由图并发调度”改成“同一条 PTZ 指令在一个串行执行器里按顺序执行”。

这样才能保证：

move -> capture
zoom -> capture
move -> zoom -> capture
find_target -> zoom -> capture







方案 1：单 PTZ 执行节点串行化
这是我建议你优先落地的方案，最稳。

改造思路
新增一个节点：
ptzToolExecutor

职责：

接收 state.toolCalls
过滤出 PTZ 工具调用
按原始顺序逐个执行
汇总 toolResults
再进入 aggregateResults
图结构改造
当前：

ptzAgent -> ptzMoveToPreset
ptzAgent -> ptzZoom
ptzAgent -> ptzCaptureFrame
ptzAgent -> ptzFindTarget
并发触发
改成：

ptzAgent -> ptzToolExecutor -> aggregateResults
具体修改点
在图里增加节点
文件：multi-agent-graph.ts (line 142)
新增：

.addNode("ptzToolExecutor", this.ptzToolExecutor.bind(this))
修改 routeFromPtz()
文件：multi-agent-graph.ts (line 682)
现在返回多个节点，改为统一返回：

return toolCalls.length > 0 ? ["ptzToolExecutor"] : ["generateResponse"];
新增执行器方法
复用你已有的 executeSingleToolCall(...)，见 multi-agent-graph.ts (line 1564)
核心逻辑：

private async ptzToolExecutor(state: GraphState) {
  const toolCalls = state.toolCalls || [];
  const results: ToolResult[] = [];

  for (const toolCall of toolCalls) {
    const start = Date.now();
    const result = await this.executeSingleToolCall(
      toolCall,
      state.userId,
      state.language,
    );
    result.executionTime = Date.now() - start;
    results.push(result);
  }

  return { toolResults: results };
}
加边
文件：multi-agent-graph.ts (line 187)
新增：

.addEdge("ptzToolExecutor", "aggregateResults")
并删除或停用 PTZ 原子节点在单任务路径上的条件路由入口。
